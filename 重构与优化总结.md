# PDF 转 Markdown 项目重构与优化总结

本文档记录了近期针对 `P2_Pdf_To_Markdown` 项目中核心数据处理流水线（OCR 数据与 PyMuPDF 数据融合、页内段落合并、跨页合并及格式化输出）的重构逻辑、Bug 修复和性能优化点。

---

## 1. 核心架构重构：扁平化五步流水线
旧版代码中存在多个分离的合并函数（如 `merge_nodes`、`merge_nodes_two`、`merge_nodes_three`），导致了严重的数据数组过度嵌套问题（如 `[[[x,y,x,y]]]`）。

**重构动作**：
参考 `7_ideal_pipeline.py` 的设计思路，在 `Fun_OCR_Pdf_To_MarkDown.py` 和主调度文件 `6_server_pdf_to_markdown_v2.py` 中引入了全新的、无嵌套的“扁平化五步流水线”：
1. **pdf-2-json**: DeepSeek OCR 识别数据（排版好，可能有错字）。
2. **pdf-2-json-python**: PyMuPDF 提取数据（文字准）。
3. **pdf-2-json-py-to-ds**: 融合 OCR 框和 PyMuPDF 文字。
4. **pdf-2-json-py-to-ds-curpage-merged**: 执行页内段落智能合并。
5. **pdf-3-mk**: 执行跨页合并及坐标转换，输出最终扁平格式化文件。

所有合并操作均采用 `.extend()` 代替 `.append()`，从源头上杜绝了多层嵌套。

---

## 2. 文本提取与融合阶段（Step 3）的深度优化

### 2.1 表格与特殊区块的 OCR 回退机制
- **问题**：旧代码中，无论是表格还是图片，只要进入合并流程，原有排版（如 HTML 结构）可能会被覆盖或丢失。
- **优化**：在 `step4_merge_ocr_and_py_json` 中，增加了对 OCR 文本的检测。如果文本中包含 `<table>` 标签，直接跳过合并，100% 信任并保留 OCR 的原始 HTML 表格数据。

### 2.2 缺失内容的 `raw_text` 兜底解析
- **问题**：如果 `PyMuPDF` 因图片层等原因没有提取到文字（`combined_py_text` 为空），原有的 `text_content` 会直接为空。
- **优化**：加入容错逻辑，当遇到没有文字的区块时，根据区块的坐标（`bbox`）去 OCR 的原始解析结构 `raw_text` 中（例如 `<|det|>[[x,y,x,y]]<|/det|>`）使用正则匹配，强行找回带有最初始排版的文本内容，确保数据不丢失。

### 2.3 保留排版用的空白字符
- **问题**：在存入 `texts` 节点时，过度使用了 `.strip()` 方法，导致段落首尾用于缩进或对齐的普通空格和全角空格丢失，破坏了后续的分词功能。
- **优化**：区分了“用于展示的最终拼装字段”和“原始痕迹字段”。在收集被合并节点存入 `texts` 数组时，保留了包含所有空格的 `original_text`。

---

## 3. 段落智能合并阶段（Step 4）的优化

### 3.1 引入智能小标题防粘连拦截器
- **问题**：类似“（一）主流加密资产：动态价值保全与实现机制”的章节标题，因为末尾没有句号等终止标点，会被旧逻辑误判为“被截断的句子”，从而错误地与下一段正文连成一段。
- **优化**：在 `test_merge_paragraphs.py` 和主合并函数中，增加了 `is_title_format()` 正则验证函数。能自动识别如 `一、`、`（一）`、`1.`、`第一章` 或 `# ` 开头的标题特征。如果被判定为标题，强制阻断其与后续正文的合并。

### 3.2 恢复 `content` 最大合并长度限制
- **问题**：新架构最初为了极简去掉了旧版的 `max_length` 逻辑，导致正文内容可能会被无限拼接得过长。
- **优化**：在格式化输出前，加入了一层二次打包过滤逻辑，按照旧版 1024 个字符的阈值，将超长段落进行合理截断，确保单个最终输出节点的文本内容长度符合下游服务（如大模型校验）的输入限制。

---

## 4. PyMuPDF 原生提取阶段（Step 2）的防重影去重

- **问题**：由于某些 PDF 制作时采用了“伪加粗”或文字阴影效果（在极近的坐标位置重复绘制多层相同的文本），导致 `PyMuPDF` 在按顺序读取时，会出现同一句话被重复读出两三遍的情况。
- **优化**：在 `pdf_to_json.py`（及测试脚本 `test_extract_pdf.py`）中，引入了基于“文本内容 + 近似 Y 坐标”的防重机制。通过建立一个 `seen_lines` 集合，计算 `(line_text.strip(), int(y/2))` 作为行签名。如果在几乎同样的高度出现了相同的文本，自动过滤抛弃，完美解决了伪加粗造成的文字重叠问题。

---

## 5. 最终数据结构输出调整

根据下游业务对数据追踪的需求，最终写入到 `pdf-3-mk/processed_merged_nodes...format.json` 中的数据结构被严格对齐到以下格式：

```json
{
    "content": "合并后的正文内容或加了Markdown前缀的标题",
    "texts": ["被合并区块的原始文本片段1", "原始片段2"],
    "page": "所在首页码",
    "pages": [["页码1", "页码2"]],
    "nodes_text_len": [[片段1长度, 片段2长度]],
    "nodes_index": [[片段1索引, 片段2索引]],
    "boxs": [[[x1,y1,x2,y2], [x3,y3,x4,y4]]],
    "image_dims": [[{"w": 原始宽, "h": 原始高}]],
    "ref": ["text", "text"],
    "id": 1,
    "type": "text",
    "number": 0
}
```

其中特别注意：
1. `boxs` 内的坐标全部统一执行了向基准 `1024` 的等比例缩放。
2. `image_dims` 保留了未经缩放前的原始页面宽高。
3. `number` 字段自动取值为第一个节点的 `nodes_index`。
